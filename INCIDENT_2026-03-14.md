## 🔴 신규 이슈 (2026-03-18) — 스케줄러 매일 `inserted=0`, 당일 데이터 미수집

### 무슨 일이 있었나
평일 16시 스케줄러(`fetchAllData`)가 실행되지만 KOSPI·KOSDAQ·금·유가·종목 전 데이터에서
`inserted=0, skipped=전체`가 반복되고 당일 데이터가 DB에 쌓이지 않음.

### 왜 발생했나
`KrxCommonUtil.getLastTradingDay()`의 로직 버그.

```java
// 수정 전 (잘못된 코드)
switch (date.getDayOfWeek()) {
    case DayOfWeek.MONDAY: return date.minusDays(3);  // 금요일 (틀림 — 월요일 16시엔 오늘이 영업일)
    case DayOfWeek.SUNDAY: return date.minusDays(2);  // 금요일 (맞음)
    default: return date.minusDays(1);                // 어제 (틀림 — 16시엔 오늘이 마감됨)
}
```

스케줄러는 `16:00`(장 마감 `15:30` 이후)에 실행되므로 당일이 가장 최근 영업일이다.
그런데 `default: minusDays(1)`이 항상 어제를 반환하므로, 어제 데이터를 재요청 → DB에 이미 있음 → 전부 `skipped`.
`MONDAY`도 `minusDays(3)` = 지난 금요일을 반환하여 월요일 당일 데이터를 놓침.

### 어떻게 고쳤나
`getLastTradingDay()`가 평일(월~금)에는 오늘을 반환하도록 수정.
토·일만 직전 금요일을 반환.

```java
// 수정 후
switch (date.getDayOfWeek()) {
    case DayOfWeek.SATURDAY: return date.minusDays(1);  // 금요일
    case DayOfWeek.SUNDAY:   return date.minusDays(2);  // 금요일
    default:                  return date;               // 월~금: 오늘
}
```

### 재발 방지
- 스케줄러 실행 시간(16시)과 장 마감 시간(15:30)을 고려한 날짜 계산 필수
- `getLastTradingDay()` 용도: "가장 최근 마감된 영업일" → 16시 이후 호출 시 오늘이 맞음

### 왜 발생했나
- `nohup java -jar ...&`으로 기동한 프로세스는 셸 세션 종료 후에도 백그라운드에서 계속 살아 있음

- 결과적으로 구 프로세스가 8080을 점유한 상태에서 신 프로세스 기동 시도 → BindException

### 어떻게 고쳤나
```bash
fuser -k 8080/tcp   # 8080 포트 점유 프로세스 강제 종료
sleep 3
nohup java -jar build/libs/backend-0.0.1-SNAPSHOT.jar > nohup.out 2>&1 &
```

### 재발 방지
- `pkill` 대신 `fuser -k 8080/tcp` 사용하여 포트 기준으로 프로세스 종료
- `deploy.yml` CI/CD 스크립트에 동일 로직 추가 필요 (STOCK_CHART_TODO.md 참조)

## 🔴 신규 이슈 (2026-03-15) — 뉴스 링크 클릭 시 네이버 뉴스 홈화면으로 이동

### 무슨 일이 있었나
크롤링된 기사 제목은 정상 표시되지만, 링크를 클릭하면 개별 기사가 아닌 네이버 뉴스 홈화면(`https://news.naver.com/` 또는 `https://finance.naver.com/news/`)이 열린다.

### 왜 발생했나

`NewsCrawler.java`의 셀렉터가 `#newsMainTop .main_news ul li a`인데, 네이버 파이낸스 페이지 구조상 해당 `<a>` 태그의 `href`가 기사 개별 URL이 아니라 **뉴스 홈 URL이나 빈 값**으로 되어 있는 경우가 있다.

`Jsoup.absUrl("href")`는 빈 href → 현재 페이지 URL, 상대경로 `/` → 도메인 루트를 반환하기 때문에 홈화면 URL이 DB에 저장된다.

```java
// NewsCrawler.java
String href = link.absUrl("href");  // ← href가 "" 또는 "/"이면 홈화면 URL 반환
if (!title.isEmpty() && !href.isEmpty()) {  // ← 홈화면 URL은 empty가 아니므로 통과
```

### 어떻게 고쳤나

href를 저장하기 전에 기사 URL 패턴 검증 필터를 추가한다.
네이버 뉴스 기사 URL은 반드시 `article` 또는 `articleid` 또는 `article_id` 키워드를 포함한다.

```java
// 수정 후
String href = link.absUrl("href");
if (href.isEmpty()) href = link.attr("href"); // absUrl 실패 시 raw href 시도
if (!title.isEmpty() && isValidArticleUrl(href)) { ... }

private boolean isValidArticleUrl(String url) {
    if (url == null || url.isEmpty()) return false;
    return url.contains("article") || url.contains("articleid") || url.contains("news_read");
}
```

### 재발 방지
- 크롤러에서 URL 저장 전 기사 URL 패턴 검증 필수
- 크롤링 결과 샘플 로그(제목 + URL)를 INFO 레벨로 출력하여 이상 즉시 감지

## 이슈 1. 프론트엔드 외부 접속 불가

### 무슨 일이 있었나
외부 브라우저에서 FinanceHub에 접속하면 화면에 데이터가 전혀 표시되지 않았다.
서버에서 직접 `curl http://localhost:80` 을 때리면 정상 응답이 오지만,
외부 PC 브라우저에서는 API 요청이 전부 실패하여 빈 화면만 렌더링됐다.

### 왜 발생했나

**원인 1 — API 주소가 localhost로 하드코딩되어 있었음**

`app/src/util/fetchRequest.ts` 에 BASE_URL이 아래와 같이 박혀 있었다.

```ts
const BASE_URL = "http://localhost:8080";
```

외부 브라우저가 이 URL로 요청을 보내면 서버가 아니라 자기 자신의 PC 8080 포트로 요청이 가므로 당연히 연결 실패.

**원인 2 — nginx proxy_pass 설정 오류**

`nginx.conf` 에 `/api/` 경로를 백엔드로 프록시하는 설정이 있었는데 trailing slash가 없었다.

```nginx
location /api/ {
    proxy_pass http://172.17.0.1:8080;   ← 잘못된 설정
}
```

nginx에서 `proxy_pass`에 trailing slash가 없으면 요청 URL의 `/api/` prefix를 제거하지 않고 그대로 백엔드에 전달한다.
즉 `/api/news/list` 요청이 백엔드에 `/api/news/list` 로 그대로 도착하는데,
백엔드 엔드포인트는 `/news/list` 이므로 404가 반환됐다.

두 문제가 겹쳐서, 원인 1을 고쳐도 원인 2 때문에 여전히 404가 났다.

### 어떻게 고쳤나

**1단계 — BASE_URL을 상대 경로로 변경** (`app/src/util/fetchRequest.ts`)

```ts
// 수정 전
const BASE_URL = "http://localhost:8080";

// 수정 후
const BASE_URL = "/api";
```

브라우저가 현재 접속한 서버의 `/api/...` 경로로 요청하게 되어, 외부에서도 동일하게 nginx를 경유한다.

**2단계 — nginx proxy_pass trailing slash 추가** (`nginx.conf`)

```nginx
# 수정 전
location /api/ {
    proxy_pass http://172.17.0.1:8080;

# 수정 후
location /api/ {
    proxy_pass http://172.17.0.1:8080/;
```

trailing slash를 추가하면 nginx가 `/api/` prefix를 제거한 뒤 백엔드로 전달한다.
예: `/api/news/list` → 백엔드 `/news/list` 로 정상 라우팅.

**3단계 — 프론트엔드 재빌드 및 nginx 재시작**

```bash
cd app && npm run build
docker restart nginx
```

PostgreSQL에 직접 접속하여 4개 컬럼 타입을 변경했다.

```sql
ALTER TABLE financehub.ipo ALTER COLUMN period TYPE VARCHAR(200);
ALTER TABLE financehub.ipo ALTER COLUMN fixed_offering_price TYPE VARCHAR(200);
ALTER TABLE financehub.ipo ALTER COLUMN expected_offering_price TYPE VARCHAR(200);
```

# 잠재 이슈 — nginx 라우팅 설정 불일치 및 미완료 항목

> 장애로 확정된 것은 아니나, 현재 설정에서 에러가 재발하거나 새로 발생할 수 있는 구조적 문제를 정리한다.

## 이슈 1. `app/nginx.conf` 미수정 — 루트 `nginx.conf`와 불일치

### 무슨 문제인가

`docker-compose.yml`은 루트 `./nginx.conf`를 마운트해서 사용한다.

```yaml
volumes:
  - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
```

INCIDENT 이슈 1에서 루트 `nginx.conf`의 trailing slash를 추가해 수정했지만,
`app/nginx.conf`는 수정되지 않아 두 파일이 다른 상태로 방치되어 있다.

| 파일 | proxy_pass 설정 | 상태 |
|------|----------------|------|
| `nginx.conf` (루트) | `proxy_pass http://172.17.0.1:8080/;` | ✅ 정상 |
| `app/nginx.conf` | `proxy_pass http://172.17.0.1:8080;` | ❌ trailing slash 없음 |

### 왜 문제인가

현재는 루트 `nginx.conf`를 사용하므로 운영 중 즉각적인 장애는 없다.
그러나 아래 상황에서 모든 API 요청이 404로 실패한다.

- Dockerfile 작성 시 `app/nginx.conf`를 COPY해서 사용하는 경우
- 누군가 `app/nginx.conf`를 기준으로 설정을 수정하는 경우

`app/nginx.conf`를 사용하면 `/api/ipo/list` → 백엔드에 `/api/ipo/list`로 전달 →
백엔드 엔드포인트는 `/ipo/list`이므로 404.

### 조치 방향

`app/nginx.conf`를 루트 `nginx.conf`와 동일하게 수정하거나, `app/nginx.conf`를 삭제하여 혼란을 방지한다.

## 이슈 2. SPA 라우팅(`/ipo`, `/news`, `/stocks` 직접 접근) nginx 의존

### 무슨 문제인가

Vue Router가 `createWebHistory()` (HTML5 History Mode)를 사용한다.
브라우저에서 `/ipo`, `/news`, `/stocks`를 직접 입력하거나 새로고침하면
nginx가 해당 경로의 파일을 찾다가 없으면 `index.html`을 반환해야 한다.

현재 설정:
```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

이 설정이 있으므로 현재는 동작한다. 그러나 아래 상황에서 404가 발생한다.

- `try_files` 지시어가 변경·삭제되는 경우
- nginx 앞에 별도 리버스 프록시(AWS ALB, Cloudflare 등)가 추가될 때 해당 레이어에서 경로를 처리하는 경우

### 조치 방향

nginx 설정 변경 시 `try_files $uri $uri/ /index.html;` 라인을 반드시 유지한다.
SPA 라우팅 경로(`/ipo`, `/news`, `/stocks`)를 주석으로 명시하여 의도를 문서화한다.

## 이슈 3. `/admin` 엔드포인트 nginx 레벨 접근 제어 없음

### 무슨 문제인가

`SchedulerController`의 엔드포인트들이 nginx에서 별도 보호 없이 외부에 노출된다.

```
POST /api/admin/run-all    — 전체 크롤링 수동 트리거 (Chromium 포함)
POST /api/admin/run-krx    — KRX 데이터 수동 수집
GET  /api/admin/scheduler-logs — 실행 로그 조회
```

`/api/` 경로 전체가 백엔드로 프록시되기 때문에, 위 경로를 아는 사람은 누구나
외부에서 크롤링을 반복 트리거하여 서버에 과부하를 줄 수 있다.

### 조치 방향

nginx에서 `/api/admin/` 경로에 대해 허용 IP를 제한하거나 기본 인증(Basic Auth)을 추가한다.

```nginx
location /api/admin/ {
    allow 127.0.0.1;
    allow <배포 서버 IP>;
    deny all;
    proxy_pass http://172.17.0.1:8080/admin/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}
```

### 무슨 문제인가

현재 nginx는 `/api/` 단일 location으로 모든 백엔드 요청을 처리한다.

```nginx
location /api/ {
    proxy_pass http://172.17.0.1:8080/;
}
```

백엔드 엔드포인트별 경로 구조:

| 프론트 호출 경로 | nginx 처리 | 백엔드 도달 경로 | 상태 |
|----------------|-----------|----------------|------|
| `/api/dashboard/data` | `/api/` 매칭 | `/dashboard/data` | ✅ |
| `/api/ipo/list` | `/api/` 매칭 | `/ipo/list` | ✅ |
| `/api/news/list` | `/api/` 매칭 | `/news/list` | ✅ |
| `/api/admin/run-krx` | `/api/` 매칭 | `/admin/run-krx` | ✅ (단, 보호 없음) |

현재 기능상 동작은 하나, `/api/` 하나로 모든 경로를 처리하므로
경로별 접근 제어, 타임아웃, 캐싱 등 세분화된 설정이 불가능하다.
